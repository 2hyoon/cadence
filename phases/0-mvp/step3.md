## Task

Create the state layer in `store/`: a pure reducer + a single React Context provider with client-only hydration.

1. `store/reducer.ts` — a **pure** `reducer(state: AppState, action: Action): AppState`. No side effects, no I/O, no `Date.now()`/`crypto` inside the reducer for values that affect equality testing — pass such values **in the action payload** instead (so the reducer is deterministic and unit-testable).
   - `Action` union covering at least: `add`, `edit`, `toggle` (toggle `completed`), `delete`, and a `hydrate`/`replace` action that loads a full `AppState`.
   - `add` receives a fully-formed `Todo` (id, createdAt generated by the caller) in the payload.
   - `delete` of a parent also removes its subtasks (children with matching `parentId`). (Subtask creation UI comes later; handle the cascade now so deletion is correct once subtasks exist.)
2. `store/TodoProvider.tsx` (Client Component, `"use client"`) — holds `useReducer(reducer, emptyState())`, exposes state + dispatch (or thin action creators) via Context, and:
   - **Client-only hydration**: on mount (`useEffect`), call `loadState()` and dispatch `hydrate`. Do not read storage during render. Reason: SSR/client markup must match.
   - **Persist on change**: in an effect that depends on state, call `saveState(state)`. Skip persisting the initial pre-hydration render.
   - Expose a `hydrated` boolean so the UI can show skeleton/empty before load (used in step 4).
3. A `useTodos()` hook to consume the context (throws if used outside the provider).

Action creators may live in `store/` and are the place to call `crypto.randomUUID()` and `new Date().toISOString()` before dispatching — keeping the reducer pure.

Unit tests for the **reducer** (pure, no React): add/edit/toggle/delete, parent delete cascades to children, hydrate replaces state. These are prime TDD targets.

## Verification Procedure

1. Run the AC commands above.
2. Architecture checklist: reducer is pure (no I/O, deterministic); provider is a Client Component; only `storage.ts` touches localStorage; hydration is client-only.
3. Update step 3 in `phases/0-mvp/index.json` (completed + summary: `Action` variants, hook name `useTodos`, `hydrated` flag, persist boundary).

## Prohibitions

- Don't call `localStorage` directly from the store. Reason: must go through `persistence/storage.ts` (CLAUDE.md).
- Don't generate ids/timestamps inside the reducer. Reason: it must be pure and deterministic for tests (ADR-004).
- Don't read storage during render. Reason: causes SSR/client hydration mismatch.
- Don't break existing tests.
